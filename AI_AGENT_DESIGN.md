# AI 双智能体工作流设计草案

本文档总结了我们讨论的“调度 AI + 解答 AI”方案，并给出未来落地时的设计要点。目标是在不牺牲安全性的前提下，让 AI 插件拥有更强的自主调度能力和可扩展性。

---

## 1. 架构概览

```
┌────────────┐          ┌────────────────────┐          ┌──────────────┐
│  用户输入   │ ───────▶ │  调度 AI（Coordinator） │ ───────▶ │  解答 AI（Executor） │
└────────────┘          └────────────────────┘          └──────────────┘
         ▲                         │                              │
         │                         ▼                              ▼
         │                ┌────────────────┐              ┌────────────────┐
         └────────────────│ 前端执行控制层 │◀──────────── │   工具 / 文档 API  │
                          └────────────────┘              └────────────────┘
```

- **调度 AI**：负责理解用户目标、维护工作计划、决定下一步操作。它不会直接改动文档，而是输出结构化指令，由前端执行或转交解答 AI。
- **解答 AI**：专注处理“生成回答/内容”类任务，例如总结、润色、写作。调度 AI 为其准备上下文和提示词。
- **执行控制层**：驻留在前端，暴露安全受限的工具（读文档、写文档、追加文本等）。只接受经校验的结构化指令，保证系统可控。

---

## 2. 调度 AI 职责

1. **任务分析**  
   - 解析用户意图，识别是否涉及文档读取、编辑、生成等动作。  
   - 维护一个简要的“todo 队列”（例如：阅读前 40 行、补充上下文、请求解答 AI）。

2. **上下文拼装**  
   - 通过行动指令请求前端读取文档任意范围。  
  - 可以合并多个片段、生成摘要或高亮重点，供解答 AI 使用。

3. **流程控制**  
   - 决定何时需要追加更多上下文，或何时将任务交给解答 AI。  
   - 负责判断答复是否满足用户需求，若不满足可重新规划。

4. **安全约束**  
   - 输出必须符合我们定义的 JSON Schema。  
   - 不允许直接执行写操作，只能发出指令（例如 `{"action":"append_to_document","payload":{...}}`），由前端验证并执行。

---

## 3. 解答 AI 职责

1. **专注生成**  
   - 根据调度 AI 提供的上下文和提示词生成最终回答。  
   - 返回纯文本或结构化结果，由前端渲染给用户。

2. **轻量协议**  
   - 解答 AI 不与工具层直接交互，所有上下文都由调度 AI 预处理。  
   - 若发现信息不足，可以返回 `MORE_CONTEXT_NEEDED` 让调度 AI 再次补充。

---

## 4. 结构化指令设计

调度 AI 输出的指令建议使用 JSON，对应以下动作类型（可按需扩展）：

| action | 描述 | 关键字段 |
| --- | --- | --- |
| `read_document` | 请求读取文档某个范围 | `range: { startLine, endLine }`；若缺省则按默认块大小读取 |
| `summarize_range` | 让前端生成某范围摘要以节省 token | `range`, `targetLength` |
| `delegate_to_executor` | 将任务交给解答 AI | `prompt`, `context`, `expectedFormat` |
| `append_to_document` | 在文末追加内容（需经过用户确认或安全策略） | `content`, `preview` |
| `replace_range` | 替换特定区间的文本 | `range`, `content`, `justification` |
| `finish` | 结束流程并返回最终结果 | `answer`, `metadata` |

指令校验失败或违反策略时，前端应拒绝执行并向调度 AI 返回错误说明。

---

## 5. 前端执行控制层

1. **安全沙箱**  
   - 所有写操作需要满足用户设定的策略（例如：显式确认、只允许特定范围）。  
   - 读取操作应限制最大行数/字符数，避免一次性加载超大文件。

2. **状态管理**  
   - 维护当前文档缓存、已读取范围、调度会话 ID 等，供两层 AI 使用。  
   - 记录每一步的执行日志，方便排查问题。

3. **回退机制**  
   - 若调度 AI 多次给出无效指令，可终止会话并提示用户。  
   - 在必要时回退到默认的单智能体流程，以保证体验不中断。

---

## 6. 实施路线图

1. **阶段一：搭建基础框架**  
   - 定义指令 Schema、前端执行器、调度 AI 提示词模板。  
   - 在当前 `AiSidebar` 中新增一个工作模式开关以便灰度发布。

2. **阶段二：实现核心行动**  
   - 支持 `read_document`、`delegate_to_executor`、`finish` 三个基础动作。  
   - 调度 AI 只负责分批读取 + 交给解答 AI，验证流程可靠性。

3. **阶段三：扩展能力**  
   - 加入 `append_to_document`、`replace_range` 等写操作。  
   - 引入摘要/关键词工具，帮助调度 AI 提升上下文利用率。  
   - 根据用户反馈迭代提示词与安全策略。

4. **阶段四：监控与优化**  
   - 记录每次协作的 action 序列、执行耗时、失败原因。  
   - 分析常见任务（如总结、润色、代码生成）的表现，优化调度逻辑。

---

## 7. 与现有实现的差异

| 项目 | 当前版本 | 双智能体方案 |
| --- | --- | --- |
| 调度方式 | 前端硬编码 chunk 逻辑 | 调度 AI 根据 action 决定读取/执行 |
| 上下文拼装 | 固定的 prompt 模板 | 调度 AI 动态组装，支持多段组合 |
| 写操作 | 直接调用 `applyDocumentUpdate` | 需经过指令校验，可插入确认环节 |
| 可扩展性 | 修改逻辑需改代码 | 增加 action/工具即可复用 |
| 安全性 | 可控但僵化 | 仍由前端把关，且具备智能调度能力 |

---

## 8. 后续行动建议

1. 与产品/安全评审讨论默认权限与确认流程，确保新增写操作不会造成误用。  
2. 设计调度 AI 的系统提示词，明确输出格式、失败处理与安全边界。  
3. 选定测试场景（如“逐段阅读新闻”、“批量生成摘要”），以驱动 MVP 实现。  
4. 实现时需将每一步调度过程（计划、指令、执行反馈）同步输出到 AI 聊天框，让用户实时跟踪流程，类似 Vibe Coding 的可视化体验。  
5. 逐步将现有逻辑迁移到指令执行框架，保留回退选项，保障线上稳定性。

---

通过以上方案，AI 插件可以在保证安全性的前提下，获得类似现代 Agent 框架的调度能力，为后续接入更多能力（搜索、分析、批量编辑等）打下基础。
